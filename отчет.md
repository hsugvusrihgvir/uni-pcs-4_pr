### Введение

##### Цель:
Разработка программы, предназначенной для автоматической генерации HTML-документации по исходному коду на языке C++ на основе специальных комментариев.
##### Область действия:
Система предназначена для обработки исходных файлов C++ (`.cpp`, `.h`) и формирования структурированной HTML-документации.

Система:
- анализирует исходный код;
- извлекает комментарии документации;
- определяет основные элементы программы;
- генерирует HTML-страницы.

Ограничения:
- используется упрощённый анализ C++;
- поддерживаются только комментарии `///` и `/* */`.
##### Определения и аббревиатуры:
- HTML (HyperText Markup Language) - язык разметки для создания веб-страниц.
-----
### Общее описание
##### Описание продукта:
Система представляет собой консольное приложение, выполняющее анализ исходного кода C++ и генерирующее HTML-документацию. Программа предназначена для студентов и разработчиков, которым необходимо быстро получать описание структуры проекта.
#### Функции продукта:
- поиск файлов `.cpp` и `.h` в указанной директории (включая вложенные папки);
- чтение содержимого файлов;
- извлечение комментариев;
- определение функций и классов;
- связывание комментариев с элементами кода;
- генерация HTML-документации;
- сохранение результата в указанную директорию.
##### Характеристики пользователей:
- Студенты (используют систему для оформления учебных проектов);
- Начинающие разработчики (используют систему для анализа структуры кода).

#### Фабричный метод

В процессе анализа кода создаётся множество объектов разных классов (`ProjectD`, `FileD`, `ClassD`, `FunctionD`).

При прямом создании объектов в модуле анализа возникает зависимость от конкретных классов модели, усложняется изменение и расширение системы и логика создания объектов распределяется по коду.

Паттерн «Фабричный метод» применяется для централизованного управления процессом создания объектов и снижения связанности между компонентами системы.

###### Как будет использоваться в проекте

Создание объектов документации выносится в отдельный класс:

```cpp
class DocFactory {  
public:  
    static FileDoc createFileDoc(...);  
    static ClassDoc createClassDoc(...);  
    static FunctionDoc createFunctionDoc(...);  
};
``` 
Модуль анализа использует фабрику для получения объектов:
```cpp
auto classDoc = DocFactory::createClassDoc(name, description);
```
Таким образом, модуль анализа не зависит от конкретных реализаций классов, а только использует интерфейс создания.

#### Одиночка

В системе присутствует основной класс, управляющий процессом генерации документации (`DocumentationGenerator`).

Создание нескольких экземпляров такого класса не требуется, так как процесс генерации выполняется централизованно, а наличие нескольких объектов может привести к дублированию логики и несогласованности состояния.

Паттерн «Одиночка» применяется для ограничения создания класса одним экземпляром и обеспечения единой точки доступа к нему.

###### Как будет использоваться в проекте

Класс реализуется таким образом, чтобы можно было получить только один его экземпляр:

```cpp
class DocumentationGenerator {  
private:  
    static DocumentationGenerator* instance;  
  
    DocumentationGenerator() {}  
  
public:  
    static DocumentationGenerator& getInstance() {  
        if (!instance)  
            instance = new DocumentationGenerator();  
        return *instance;  
    }  
};
```
Использование:
```cpp
auto& generator = DocumentationGenerator::getInstance();
```
Таким образом, в системе используется один объект, который управляет процессом генерации документации.

#### Фасад

Система включает несколько модулей, выполняющих различные этапы обработки:
- `FileScanner` (поиск файлов)
- `CodeParser` (анализ кода)
- `HtmlGenerator` (генерация HTML)

Прямое взаимодействие с каждым модулем усложняет использование системы и требует знания её внутренней структуры.

Паттерн «Фасад» применяется для предоставления единого упрощённого интерфейса к системе и сокрытия внутренней реализации.

###### Как будет использоваться в проекте

Класс `DocumentationGenerator` выступает в роли фасада и объединяет все этапы обработки:
```cpp
class DocumentationGenerator {  
public:  
    void generate(std::string inputPath, std::string outputPath) {  
        auto files = scanner.scan(inputPath);  
        auto project = parser.parse(files);  
        htmlGenerator.generate(project, outputPath);  
    }  
  
private:  
    FileScanner scanner;  
    CodeParser parser;  
    HtmlGenerator htmlGenerator;  
};
```
Использование:
```cpp
DocumentationGenerator generator;  
generator.generate(inputPath, outputPath);
```
Таким образом, взаимодействие с системой осуществляется через один метод, а детали работы отдельных модулей скрыты.

#### Стратегия

В системе генерация документации может выполняться разными способами. Например, может использоваться простая HTML-документация или HTML-документация с более подробным описанием.

При размещении всех вариантов генерации в одном классе `HtmlGenerator` код становится сложнее изменять и расширять.

Паттерн «Стратегия» применяется для вынесения разных алгоритмов генерации в отдельные классы и возможности выбирать нужный способ генерации.

###### Как будет использоваться в проекте

Создаётся общий интерфейс стратегии:
```cpp
class HtmlGenerationStrategy {  
public:  
    virtual void generate(ProjectD& project) = 0;  
};
```
Для разных вариантов генерации создаются отдельные классы:
```cpp
class SimpleHtmlStrategy : public HtmlGenerationStrategy {  
public:  
    void generate(ProjectD& project) override {  
        // генерация простой HTML-документации  
    }  
};  
  
class DetailedHtmlStrategy : public HtmlGenerationStrategy {  
public:  
    void generate(ProjectD& project) override {  
        // генерация подробной HTML-документации  
    }  
};
```
Класс `HtmlGenerator` использует выбранную стратегию:
```cpp
class HtmlGenerator {  
private:  
    HtmlGenerationStrategy* strategy;  
  
public:  
    void setStrategy(HtmlGenerationStrategy* newStrategy) {  
        strategy = newStrategy;  
    }  
  
    void generate(ProjectD& project) {  
        strategy->generate(project);  
    }  
};
```
Таким образом, способ генерации документации можно изменять без изменения основной логики работы `HtmlGenerator`.
